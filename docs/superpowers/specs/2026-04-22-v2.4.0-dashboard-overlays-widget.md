# v2.4.0 — Dashboard cosmetics + automation overlay + floating SOC widget

**Status:** draft
**Target version:** v2.4.0 (versionCode=240)
**Author:** brainstormed with user 2026-04-22

## Overview

Three independent changes bundled into one release:

1. **Dashboard cosmetics** — small tweaks to «Последние поездки» column layout and the «стоянка» card.
2. **Automation confirm overlay** — replace the system notification used for `confirmBeforeExecute` rules with a SYSTEM_ALERT_WINDOW overlay, matching the existing `OverlayNotificationManager` pattern.
3. **Floating SOC widget** — new optional persistent overlay that shows current SOC, estimated range, battery temperature and 12V voltage while BYDMate is in the background.

Items 1 and 2 are refinements of existing screens. Item 3 is a new subsystem.

## 2. Automation confirm overlay

### 2.1 Current behaviour

`AutomationEngine.showConfirmNotification` (`AutomationEngine.kt:281-321`) posts a `NotificationCompat` notification with `PRIORITY_HIGH`, two actions «Выполнить» and «Отмена», and a 30-second `setTimeoutAfter`. The notification action intents go to an internal `BroadcastReceiver` registered in `registerConfirmReceiver`.

Problems:
- System shade on DiLink is awkward to reach (small targets, easy to miss).
- Notification is easy to overlook during audio playback / navigation.
- If the user taps outside the action buttons, the notification dismisses without recording a decision.

### 2.2 New design

Keep the existing `BroadcastReceiver` and `pendingConfirmations` map. Replace the notification layer with a SYSTEM_ALERT_WINDOW overlay using the same plumbing as `OverlayNotificationManager`.

New class `ConfirmOverlayManager` (parallel to `OverlayNotificationManager`):

```kotlin
class ConfirmOverlayManager(private val context: Context) {
    fun showConfirm(
        ruleName: String,
        actionsSummary: String,
        timeoutMs: Long = 30_000L,
        onConfirm: () -> Unit,
        onCancel: () -> Unit,
    )
}
```

Responsibilities:
- Inflate a `ComposeView` (or plain Android view) via `WindowManager.addView` with `LayoutParams.TYPE_APPLICATION_OVERLAY`.
- Layout: centered card, max width ~380dp, same visual language as `CardDetailDialog` — title (rule name), body (actions summary), two buttons «Выполнить» / «Отмена».
- Fire user-chosen callback, auto-dismiss after timeout, play ringtone once (same as notification-action overlay).
- If `Settings.canDrawOverlays(context) == false`, fall back to the existing `showConfirmNotification` path so users without the permission still get a prompt.

### 2.3 Integration

In `AutomationEngine.evaluateRules`:

```kotlin
if (rule.confirmBeforeExecute) {
    if (Settings.canDrawOverlays(context)) {
        confirmOverlayManager.showConfirm(
            ruleName = rule.name,
            actionsSummary = actions.joinToString(", ") { it.displayName },
            onConfirm = { scope.launch { executeAndLog(rule, actions, snapshot, TrackingService.lastData.value) } },
            onCancel = { scope.launch { logCancelled(rule, snapshot) } },
        )
    } else {
        showConfirmNotification(rule, actions, snapshot)  // existing fallback
    }
}
```

The `pendingConfirmations` map + `BroadcastReceiver` path remains as the notification fallback. The overlay path handles its own state via callbacks — no shared pending map required.

### 2.4 Future-proofing hook

When the floating widget (item 3) is later wired to show confirmations in-place, `ConfirmOverlayManager` will expose an additional method `showConfirmInWidget(anchor: View, ...)` that the widget invokes to transform itself into a confirm state. **Not implemented in v2.4.0.** This section exists only to note the design surface.

### 2.5 File changes (Phase 2)

- **New:** `data/automation/ConfirmOverlayManager.kt` (~150 LOC).
- `AutomationEngine.kt` — inject `ConfirmOverlayManager`, branch in `evaluateRules`, keep existing notification path as fallback.
- `di/AppModule.kt` — provide `ConfirmOverlayManager` singleton.

## 3. Floating SOC widget

### 3.1 Data source

Subscribe to `TrackingService.lastData` (existing `StateFlow<DiParsData?>`). Display fields:

| Field | Source | Format |
|---|---|---|
| SOC | `data.soc` | `83%`, colored by `socColor()` |
| Range km | `estimateRangeKm(data.soc)` (existing) | `~274 км`, monospace |
| Battery temp | `data.avgBatTemp` | `16°C`, muted |
| 12V | `data.voltage12v` | `13.8 V`, muted |

Same helper functions already used by `updateNotification` in `TrackingService.kt:501-517`. No new computations.

### 3.2 Visual layout (variant B2)

Two-row mini card (~200dp × 72dp), no buttons on the card itself:

```
╭──────────────────╮
│  83%     ~274 км │
│  ─────────────── │
│  16°C    13.8 V  │
╰──────────────────╯
```

- **Background:** `CardSurface` (#1E1E22).
- **Border:** 1.5dp, color by worst status among SOC and 12V (`AccentGreen` / `SocYellow` / `SocRed`); grey α 0.4 when data is missing.
- **Corner radius:** 12dp.
- **Shadow:** 0 6dp 20dp rgba(0,0,0,0.5) — lifts the card off the background content it floats over.
- **Row 1 (top):**
  - Left: SOC percentage, 18sp, bold, monospace, color = border color.
  - Right: range `~274 км`, 14sp, `TextPrimary`, monospace.
- **Divider:** 1dp, `TextMuted` α 0.2.
- **Row 2 (bottom):**
  - Left: temperature, 11sp, `TextMuted`, monospace.
  - Right: 12V voltage, 11sp, `TextMuted`, monospace.
- **Missing data:** show `—` in place of the value; keep layout stable.

### 3.3 Interaction — drag-to-trash to hide

The widget itself stays clean — no close button. Hiding is done by dragging the widget into a trash zone that appears during drag.

| Gesture | Action |
|---|---|
| Tap (press + release within 8dp) | Launch `MainActivity` via `Intent.FLAG_ACTIVITY_NEW_TASK`. |
| Press + drag (>8dp threshold) | Enter drag mode. Widget follows finger (`WindowManager.updateViewLayout` on every `ACTION_MOVE`). The trash zone fades in at bottom-center of the screen. Widget visibly «lifts» — slight `scale(1.04)` + stronger shadow. |
| Drag over trash zone | Widget dims to α 0.5 + `scale(0.9)` + greyscale. Trash icon enlarges to `scale(1.3)` and turns solid red. Clear visual affordance «release to hide». |
| Release over trash zone | Hide widget and set `floating_widget_enabled = false` in `SharedPreferences`. `SettingsScreen` must read this flag via a `Flow<Boolean>` (or `collectAsState`) so the on-screen toggle reflects the change immediately on next open. |
| Release elsewhere | Save final `x,y` to `SharedPreferences`. Trash zone fades out. Widget returns to normal style. |

Drag threshold (8dp) prevents accidental moves on a touch-sensitive head unit.

### 3.3.1 Trash zone layout

The trash zone is a separate overlay view added to `WindowManager` only during drag (added on `ACTION_DOWN` after threshold is crossed, removed on `ACTION_UP`).

- **Position:** bottom-center of screen, 24dp from bottom edge.
- **Geometry:** 72dp circular with a soft red radial-gradient halo (alpha 0.18 → 0); inner 48dp dark disc with 1.5dp `SocRed` border α 0.6 and a light `backdrop-filter: blur(8dp)` so whatever is behind it (map, music UI) is softened.
- **Icon:** `✕`, 22sp, `SocRed` α 0.9.
- **Hint text:** below the circle, 11sp monospace, «перетащить сюда, чтобы скрыть». Turns solid red when the widget is over the zone.
- **Hit detection:** radial, centre-to-widget-centre distance `<= 48dp`. On enter/leave, apply the «over-trash» styling (dim widget + enlarge trash). Haptic feedback (`HapticFeedbackConstants.CONTEXT_CLICK`) is nice-to-have but optional — DiLink's head unit may not have a vibrator; guard with `view.performHapticFeedback(..., FLAG_IGNORE_VIEW_SETTING)` and ignore failures.
- **Fade in/out:** 200ms alpha transition. No fade during the 8dp pre-threshold grace period — the trash zone only appears once we're committed to drag mode.

### 3.4 Lifecycle

**Service:** The preference is to **avoid a second foreground service**. Implementation plan should first attempt to host the overlay directly inside `TrackingService` — add/remove the widget `ComposeView` on the existing `WindowManager` from a helper inside `TrackingService`. Only if that proves unstable (crashes, observer leaks) fall back to a dedicated `FloatingWidgetService` with `foregroundServiceType="specialUse"`. Spec file-list assumes the dedicated-service path to be safe; if the in-`TrackingService` path works, drop `FloatingWidgetService.kt` and inline the logic.

Start / stop triggers:
- `TrackingService.onStartCommand`: if `floating_widget_enabled = true` and `Settings.canDrawOverlays(context)`, attach widget.
- `SettingsScreen` toggle: on enable → request overlay permission if missing, on grant → attach; on disable → detach.

**Foreground/background tracking:** register `ActivityLifecycleCallbacks` in the existing `BYDMateApp` (`app/src/main/kotlin/com/bydmate/app/BYDMateApp.kt`). Keep a counter of resumed Activities. When counter becomes 0 → attach widget view; when ≥1 → detach widget view (service stays alive).

This means:
- BYDMate open → widget hidden.
- User presses Home → MainActivity.onStop → widget reappears at saved position.
- User taps widget → MainActivity.onStart → widget hides.
- No manual state toggling required.

### 3.5 Permission flow

On first enable:
1. User flips toggle to ON in Settings.
2. Check `Settings.canDrawOverlays(context)`.
3. If false → show dialog explaining why («Виджет показывается поверх других окон — нужно разрешение»), launch `Intent(ACTION_MANAGE_OVERLAY_PERMISSION)`.
4. On return from Settings activity, re-check permission. If granted → start service. If denied → revert toggle, show toast.

## Phases (for implementation plan)

The implementation plan will split this spec into three phases, committable and testable separately:

1. **Phase 1 — Dashboard cosmetics.** Two-file change. Quick verification on DiLink by inspecting «Последние поездки». No new data sources. Low risk.
2. **Phase 2 — Confirm overlay.** New `ConfirmOverlayManager` + integration in `AutomationEngine`. Tested by creating a test rule with `confirmBeforeExecute = true` on a trivial trigger (e.g. 12V > 12). Fallback path still works for users without overlay permission.
3. **Phase 3 — Floating widget.** New service + Compose view + Settings UI + lifecycle wiring. Largest piece. Test path: toggle on → permission flow → widget appears → drag → home → tap → position persists across app restart.

Each phase ends with a build on DiLink and a sanity check before moving on.

## Out of scope (explicitly)

- Changing the meaning of the «стоянка» card (item 0b/0c was discussed and rejected).
- Long-press actions on the floating widget.
- Edge-snap behaviour for the widget (user asked for free positioning).
- Close button (✕) on the widget itself — visually cluttered the card; hiding is instead done by dragging the widget into the bottom-center trash zone that appears during drag.
- In-widget quick actions (close windows, toggle climate). Deferred indefinitely.
- Showing confirm dialog inside the floating widget. Architectural note only; not implemented in v2.4.0.

## Open questions / risks

- **DiLink overlay quirks.** `OverlayNotificationManager` already works on DiLink, so the underlying WindowManager path is known good. Worst case for Phase 3: widget flicker on orientation change (but DiLink is locked landscape, so unlikely).
- **Service type on Android 12 (API 32 / DiLink).** `TrackingService` is already a foreground service — adding a second one may require a specific `foregroundServiceType`. To verify during Phase 3 implementation: whether the widget must live in its own service, or can share `TrackingService`'s foreground declaration via a simple `WindowManager.addView`/`removeView` pair invoked from `TrackingService` itself. Preference is to avoid a second service if possible.
- **Composition lifecycle.** Embedding Compose in WindowManager requires a `ViewTreeLifecycleOwner` and `ViewTreeSavedStateRegistryOwner`. The medium article linked during brainstorming covers this — implementation plan must call out the boilerplate explicitly to avoid «Compose view not attached to a ViewTreeLifecycleOwner» crashes.
